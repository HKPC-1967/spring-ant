## 3. Framework Design philosophy 
- This framework is designed to be lightweight, native, and easily extensible. It's well-structured and tested, with core features like API, error codes, and security defined. An intern can quickly get started and write maintainable business code under the supervision of an experienced developer. We mainly implemented the core features and integration with Ant Design Pro, and you can further extend and customize it based on your needs. Once the framework is set up, the rest is just business code.  
- This Spring Ant Family is best suited for full stack developers, this is the reason we removed the Data Mock part from the original Ant Design Pro, as this extra step makes our development speed slower in our full stack development practice.  

## 4. Object Types 
- DTO (Data Transfer Object/ Request Object).   
  Purpose: Used to transfer data between different parts of a system, such as between client and server.   
  We use it as the HTTP POST request body Mapping Model.

- VO (View Object / Response Object).  
  Purpose: Used as the HTTP response object returned by Controller to frontend. It should only contain fields that are safe and necessary for API output.

- Model (MyBatis Model / Database Object).  
  Purpose: Generated by MyBatis Generator and used for database interaction (DAO/Mapper layer). It represents database table structure and should not be exposed to frontend directly.  

#### Controller response VO filtering | Controller响应VO过滤
Do one extra conversion step before returning data from Controller. Convert DB Model (`Contact`) into response VO (`ContactVO`), and only keep fields that should be exposed to frontend.  

```java
// Useful Code Snippets 
@GetMapping("/contacts")
public List<ContactVO> listContacts() {
  List<Contact> contacts = contactRepository.findAllWithCompany();

  // Extra layer at Controller return boundary: filter and reshape response fields
  return contacts.stream().map(this::toContactRecord).toList();
}

private ContactVO toContactRecord(Contact contact) {
  return new ContactVO(
          contact.getId(),
          contact.getFirstName(),
          contact.getLastName(),
          contact.getEmail(),
          new CompanyRecord(
                  contact.getCompany().getId(),
                  contact.getCompany().getName()
          )
  );
}
```
- Why this is important：
  - Hide sensitive fields (e.g. password hash, phone, id number, internal status).  
  - Keep API stable even when database schema changes.  
  - Avoid coupling frontend contract to persistence model.  
- Rule of thumb: Model for DB read/write, DTO/VO for API request/response. Controller is the clearest "inbound/outbound filter point".  
- But you can still choose to define the returned fields directly in SQL if that better fits your business needs and speeds up development.
