# Code Convention 代码规范
We don't have strick Code Convention for this Open Source Project. you can define your own convention that fits for your project. Below are some conventions or tools that we used, you can refer to it if you want.  
我们没有严格的代码规范，您可以定义适合您项目的代码规范。 以下是我们有使用的一些规范，供参考。

### 0. IDEA plugins IDEA插件
We recommend installing the following plugins in IDEA.
- MybatisX
- Lombok

### 1. Prefer Multi-line annotation for Class and Field
Prefer multi-line annotation for class and field, so that when you hover over the class and field in IDEA, you can see the annotation.
类和字段上的建议用這種多行註釋，从而IDEA可以识别，鼠標悬浮在引用的类和字段上時,就可以看到其註釋了。
```java
/**
 *
 */
```

### 2. For the same field, define it in one place, do not repeat the same field in different places. Please strickly follow this for maintainability.
同样的字段，要在一处定义，不要在不同的地方重复定义同样的字段。请严格遵守这一点以便维护。

### 3. Not following the full Restful Style, only use POST (and GET, if needed) for simplicity
For GET, Please use "?a=6&b=6", not "/a/6/b/6", Please check [MainAspect.java](./../src/main/java/org/hkpc/dtd/common/core/aspect/MainAspect.java) for code implementation details.  

未严格实现Restful风格原因: 
1. 內部API並不需要restful,還更安全防止被黑客猜路徑 
2. 統一全是post,簡化integrate前端開發，後端aspect和controller, swagger 
3. 很多程序員也用不對Restful，不如统一POST簡單點

Reasons for not strictly following the full Restful style:
1. Internal APIs do not require RESTful style and using customized POST URL can be more secure to prevent hackers from guessing paths. 
2. Using POST uniformly simplifies integration with front-end development, back-end aspects, controllers, and Swagger. 
3. Many developers may not use RESTful style correctly, so it's better to keep it simple.

### 4. Method naming convention for controller: verb + noun (similar to the name generated by MyBatis Generator)
Controller方法名命名規則：動詞 + 名詞 (類似MyBatis Generator生成的名字)

CRUD (CREATE, READ, UPDATE and DELETE)

* READ (no side effects (just query, no data modification)) </br>
  READ (沒有副作用（只是查詢, 不會修改數據）) </br>
    * Single selectByPrimaryKey, selectByXxx
    * Multi selectAll, selectAllByTime, selectAllByXxx, selectAllXxxByXxx
    * Page selectPage, selectPageByXxx, selectXxxPageByXxx
  
* CREATE (has side effects, will modify data, pay attention to idempotent logic in code (i.e. whether the impact on the server is the same for one or multiple requests from the front end)) </br>
  CREATE (有副作用，會修改數據, code注意冪等邏輯(即前端請求一次和多次對服務器的影響是否相同)) </br>
    * Single insert, insertXxx, insertSelective (Not recommended, reason: 一般不推薦这种通用的business业务层面的HTTP insert接口，否则某个business变化要改时，会影响其他的business。推荐對外不同business暴露不同的business接口。)
    * Multi batchInsert, batchInsertXxx
  
* UPDATE (has side effects, will modify data, generally idempotent) </br>
  UPDATE (有副作用，會修改數據,一般冪等) </br>
    * Single updateByPrimaryKey, updateByXxx, updateByPrimaryKeySelective (Not recommended, same reason as insertSelective)
    * Multi batchUpdateByXxx
* DELETE (has side effects, will modify data, generally idempotent) </br>
  DELETE (有副作用，會修改數據,一般冪等) </br>
    * Single deleteByPrimaryKey, deleteByXxx
    * Single batchDeleteByXxx
  
* Others, named by business </br>
    * xxxSchedule

### 5. Git commit message convention  
| English                                                         | 中文                                          |
|-----------------------------------------------------------------|---------------------------------------------|
| We recommend the same commit message convention used in Ant Design Pro | 我们建议用同 Ant Design Pro 一样的 git commit 提交信息规范 |
| Common Conventional Commits types include:                      | 常见的 git commit 类型包括：                        |
| `feat`: New feature                                             | `feat`：新功能                                  |
| `fix`: Bug fix                                                  | `fix`：修复 bug                                |
| `docs`: Documentation change                                    | `docs`：文档变更                                 |
| `style`: Code style change (no behavior change)                 | `style`：代码格式（不影响功能，例如空格、分号等）                |
| `refactor`: Code refactor (no new feature, no bug fix)          | `refactor`：代码重构（既不是新增功能，也不是修复 bug）          |
| `perf`: Performance improvement                                 | `perf`：性能优化                                 |
| `test`: Add or update tests                                     | `test`：增加或修改测试                              |
| `build`: Build system or external dependency changes            | `build`：影响构建系统或外部依赖的更改（如 npm、gulp、webpack）  |
| `ci`: CI configuration or script changes                        | `ci`：持续集成相关配置或脚本的更改                         |
| `chore`: Other changes that do not modify src or test files     | `chore`：其他不属于上述类型的更改（如构建流程、辅助工具、依赖库升级等）     |
| `revert`: Revert a previous commit                              | `revert`：回滚某个更早之前的提交                        |
